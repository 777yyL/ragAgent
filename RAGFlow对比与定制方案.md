## 三、简版定制开发方案

采用**"RAGFlow 内核 + 外挂增强服务 + 上层业务应用"**三层架构，避免侵入式修改 RAGFlow 核心，降低升级成本。

### 架构分层

```
┌─────────────────────────────────────────────────┐
│  上层业务应用（新建，基于 RAGFlow API 封装）      │
│  门户 / 审核 / 审批 / 权限 / 统计分析 / 7大智能体 │
├─────────────────────────────────────────────────┤
│  增强服务层（新建，微服务）                        │
│  采集 Connector / 清洗管道 / 融合 / 补全 / AI审核 │
├─────────────────────────────────────────────────┤
│  RAGFlow 内核（原样保留）                         │
│  解析 / 切分 / 向量化 / 图谱 / 检索 / QA增强      │
└─────────────────────────────────────────────────┘
```

### 关键定制点方案

| 定制项 | 方案 | 落地路径 |
|-------|------|---------|
| 数据库/API/消息队列/爬虫/日志 | 独立"采集服务"，统一输出 JSON，调用 RAGFlow **Dataset/Document Upload API** 写入 | Debezium(CDC) + 自研 Connector + Filebeat/Fluentd + 调度器（XXL-Job/Celery） |
| 清洗管道 | 在采集服务内实现责任链，清洗后再喂给 RAGFlow | 复用 RAGFlow 预处理 + 外部管道补充 OCR 纠错 |
| 知识融合/补全/AI审核 | 独立"质量服务"，定时扫描 RAGFlow chunk（通过 API 拉取），产出建议写回业务库 | LLM 调用 + 向量相似度比对 + 人工确认工作台 |
| 人工审核/审批流程 | 上层应用封装"审核工作台"，审核通过后才调用 RAGFlow API 启用 chunk | 流程引擎（Flowable/Camunda）+ 状态机 + 三级审核模型 |
| 细粒度权限 | 上层应用做 RBAC 网关，按用户身份过滤 RAGFlow 的 dataset/chunk 访问 | 业务侧维护权限表 + 请求拦截代理 + 文档级权限 + 审计日志（等保三级） |
| 版本管理 | 利用 RAGFlow 多 dataset 天然隔离"线上/测试"版本；回滚=切换 dataset 指针 | 配置中心管理版本映射 |
| Neo4j 图谱 | 原始方案强约束 Neo4j/NebulaGraph：旁路抽取实体关系（复用 RAGFlow 图谱抽取 Prompt）写入 Neo4j；检索时双路融合 | 评估 RAGFlow 内置图是否可替代，能替代则省去 |
| 门户与智能体 | 前端新建门户，调用 RAGFlow 对话/检索 API；7 大智能体基于 Agent 工作流编排 | Vue3 + Element Plus + RAGFlow Agent |

### 写入收敛点

所有定制采集/清洗/融合的产出，**统一收敛到 RAGFlow 的 Dataset Upload API**，把 RAGFlow 当作"知识存储+检索内核"，不触碰其向量库与图谱内部结构。

### 主要风险

1. **CDC 实时同步**的稳定性与运维成本（Debezium + Kafka 链路长）。
2. **细粒度权限代理**对检索性能的影响（需在代理层做 chunk 级过滤，建议预计算权限索引）；需同时满足文档级权限 + 操作审计 100% + 等保三级。
3. **知识融合/补全/AI 审核**依赖 LLM 调用量大，成本与误报率需平衡（建议高置信自动、低置信人工）；需满足知识准确率≥99%（第195行）。
4. **Neo4j 与 RAGFlow 内置图双写一致性**，原始方案明确 Neo4j/NebulaGraph 强约束，建议优先评估内置图能否替代，规避双写。
5. **性能达标风险**：检索≤1s、问答≤5s、并发≥300 需压测验证（第192-194行）。
6. **信创适配风险**：RAGFlow 在麒麟/鲲鹏/达梦等国产环境兼容性需专项验证（第131行）。

### 结论

RAGFlow 可作为**知识处理与检索内核**承载约 40% 原生满足的需求（解析/切分/向量化/图谱/检索/标注/质量调试），且与原始方案技术路线高度一致（原始方案第99、129行直接指定 RAGFlow DeepDoc 引擎与框架）；剩余 60% 通过**外挂采集 + 上层业务应用**定制开发，技术路径清晰、风险可控。

建议优先级：**采集层 > 权限与审核 > 质量治理 > 门户与智能体 > Neo4j（强约束，需专项评估）**。
